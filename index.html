Tasy EMR • Gestão Hospitalar

🏥 Extração de Biblioteca de Fármacos para Bombas de Infusão

1. O que é o projeto

Desenvolvimento de uma consulta SQL avançada para a extração de dados técnicos de medicamentos (doses, diluições e estabilidades) do ERP Tasy. O objetivo foi fornecer a base de dados necessária para a configuração da Biblioteca de Fármacos (Drug Library) das novas bombas de infusão da marca B Braun na unidade hospitalar HMG.

2. O Problema Técnico

  • Dados Dispersos: As informações necessárias estavam espalhadas em mais de 9 tabelas diferentes.
  • Hierarquia de Dados: Muitas doses padrão estavam cadastradas na "capa" do material, mas não na aba de "Prescrição", ou vice-versa, resultando em campos vazios no relatório inicial.
  • Variações de Via: Medicamentos com vias específicas (ex: IVA, IVN) não estavam sendo vinculados corretamente às regras gerais de estabilidade.
  • Codificação Técnica: Campos retornavam apenas siglas internas (ex: 'H', 'MI'), ilegíveis para a equipe assistencial.

3. A Solução Aplicada

Utilizei técnicas avançadas de SQL para normalizar e consolidar os dados: Joins Complexos (MATERIAL, MATERIAL_DILUICAO, etc), Lógica de Fallback com NVL e NULLIF, e Normalização com DECODE e CASE para tradução de códigos para texto por extenso.

4. O Impacto na Ponta

Segurança do Paciente: Garantiu configuração exata das bombas, reduzindo riscos de erros de medicação. Redução de Retrabalho: Automação substituiu conferência manual de centenas de itens.

🛠️ Tecnologias Utilizadas:
Oracle DatabaseSQL (PL/SQL)Tasy EMR

🚀 Otimização e Correção: Relatório de Consumo Diário por Local de Estoque (Tasy EMR)

1. O que é o projeto?

Reestruturação completa do relatório de Consumo Diário por Local de Estoque (WATE-1570) no sistema Tasy EMR para parametrização de cotas e reposição inteligente de insumos.

2. O Problema Técnico

  • Dados Incorretos: Devido a um erro de Join (produto cartesiano), os valores de consumo eram multiplicados indevidamente.
  • Excesso de "Ruído": O relatório listava todos os itens do estoque, gerando documentos de 74 páginas (90% consumo zero).
  • Filtros Ineficientes: O parâmetro de data não respeitava a seleção do usuário e mostrava locais de estoque duplicados de outras unidades.

3. A Solução Aplicada

Refatoração do SQL corrigindo ligações entre as tabelas de material e movimento. Apliquei HAVING SUM(qt_consumo) > 0 para filtrar lixo e utilizei a variável :nr_estab_v para filtrar dinamicamente a unidade hospitalar atual (HMG). Formatei campos usando TO_CHAR e DISTINCT para listas limpas.

4. O Impacto na Ponta

Redução de Desperdício: O relatório passou de 74 para 8 páginas de dados úteis. Precisão Decisória: Supervisão consegue evitar falta ou excesso de medicamentos nos setores.

Tasy EMRSQL (Oracle PL/SQL)Gerenciador de Relatórios

🚀 Customização de Regras de Negócio: Controle de Requisicão de Materiais

1. O que é o projeto

Implementação de uma trava lógica no módulo de Requisição de Materiais do ERP Tasy para condicionar a obrigatoriedade de campos específicos.

2. O Problema Técnico

O campo Atendimento era opcional. Quando a operação era "Consumo Paciente", o sistema permitia salvar sem vincular um número de atendimento. Isso causava perda financeira: o item saía do estoque, mas não era faturado para o paciente.

3. A Solução Aplicada

Criação de uma Regra de Atributo no campo NR_ATENDIMENTO (Tabela REQUISICAO_MATERIAL) via dicionário de dados do Tasy:

AND :CD_OPERACAO_ESTOQUE = [ID_CONSUMO_PACIENTE]

O ícone de obrigatoriedade aparece dinamicamente impedindo o salvamento se o campo estiver vazio.

4. O Impacto na Ponta

Garantia de Receita: Eliminou a perda financeira de materiais consumidos e não cobrados. Rastreabilidade: Assegurou 100% de integridade nos itens de consumo.

🛠️ Tecnologias Utilizadas:
Oracle SQLDicionário de Dados Tasy

📊 Relatório de Eficiência e SLA Transfusional (Hemoterapia)

1. O que é o projeto

Desenvolvimento de relatório analítico de SLA para monitorar o ciclo de vida de uma transfusão, desde a solicitação médica até a beira leito (ADEP).

2. O Problema Técnico

  • Fragmentação: Dados espalhados em módulos Médico, Laboratório e Enfermagem.
  • Dados Incompletos: Registros de coleta apresentavam campos nulos.
  • Complexidade: Uma reserva pode conter múltiplas bolsas e IDs numéricos de difícil interpretação.

3. A Solução Aplicada

Desenvolvi query avançada com LEFT JOINs entre SAN_RESERVA, SAN_PRODUCAO_V e SAN_TRANSFUSAO. Usei NVL para buscar coleta em tabelas secundárias e CASE WHEN para traduzir domínios de urgência. Criei colunas calculadas para tempo de resposta em minutos.

4. O Impacto na Ponta

Identificação de Gargalos: Monitoramento exato do tempo de resposta por setor. Segurança: Controle de tempos críticos em transfusões de emergência.

🛠️ Tecnologias Utilizadas:
Oracle DatabaseSQL (DML/DQL)Tasy EMR

🚀 Otimização de Etiquetas de Dispensação por Lote

1. O que é o projeto?

Melhoria de etiquetas térmicas para farmácia hospitalar, transformando-as em ferramentas de segurança clínica.

2. O Problema Técnico

A etiqueta original não exibia horários de administração, dificultando o trabalho da enfermagem na beira-leito. Além disso, o layout causava borrões no código de barras, impedindo a leitura por scanners.

3. A Solução Aplicada

Implementei SQL Dinâmico com LISTAGG para agrupar horários em uma única linha. Reprogramei o layout em linguagem EPL (Zebra), redimensionando o barcode e reorganizando coordenadas X/Y para evitar sobreposições.

4. O Impacto na Ponta

Segurança: Enfermagem visualiza o horário exato vinculado ao lote. Eficiência: Aumento de 100% na taxa de leitura dos códigos de barras pelos coletores.

🚀 Otimização de Relatório de Dispensação Farmacêutica (CATE1011)

1. O que é o projeto

Otimização do relatório CATE1011 para conferência de medicamentos à beira do leito.

2. O Problema Técnico

  • Desperdício: Gerava uma nova página para cada lote (5 lotes = 5 folhas de papel).
  • Rastreabilidade: Não havia código de barras individual por linha de medicamento.

3. A Solução Aplicada

Alterei a lógica de agrupamento de Lote para Atendimento (nr_atendimento). Inseri componentes de barcode CODE128_C diretamente na banda de detalhes e otimizei o layout para eliminar "buracos" brancos no fim da página.

4. Impacto na Ponta

Economia: Redução de 60% no consumo de papel e toner. Agilidade: Bipagem correta de cada item na beira-leito eliminando erros manuais.
