🏥 Extração de Biblioteca de Fármacos para Bombas de Infusão

1. O que é o projeto

Desenvolvimento de uma consulta SQL avançada para a extração de dados técnicos de medicamentos (doses, diluições e estabilidades) do ERP Tasy. O objetivo foi fornecer a base de dados necessária para a configuração da Biblioteca de Fármacos (Drug Library) das novas bombas de infusão da marca B Braun na unidade hospitalar HMG.

2. O Problema Técnico

O desafio principal foi a fragmentação e inconsistência dos dados dentro do banco de dados Oracle do Tasy:

  • Dados Dispersos: As informações necessárias estavam espalhadas em mais de 9 tabelas diferentes.
  • Hierarquia de Dados: Muitas doses padrão estavam cadastradas na "capa" do material, mas não na aba de "Prescrição", ou vice-versa.
  • Variações de Via: Medicamentos com vias específicas (ex: IVA, IVN) não estavam vinculados corretamente.
  • Codificação Técnica: Campos retornavam apenas siglas internas (ex: 'H', 'MI'), ilegíveis para a parametrização do equipamento.

3. A Solução Aplicada

Utilizei técnicas avançadas de SQL para normalizar e consolidar os dados:

  • Joins Complexos: Múltiplos LEFT JOINs conectando tabelas de materiais, vias, diluições e prescrições.
  • Lógica de Fallback: Uso de funções NVL e NULLIF para buscar a dose na capa caso não existisse na aba de prescrição.
  • Normalização: Tradução de códigos internos via DECODE e CASE (ex: 'MI' para 'Minutos').
  • Filtros de Negócio: Captura de todas as variações de vias intravenosas via LIKE.

4. O Impacto na Ponta

Segurança do Paciente: Configuração exata das bombas, reduzindo riscos de erros de medicação. Redução de Retrabalho: Automação substituiu conferência manual, economizando dezenas de horas de trabalho.

🛠️ Tecnologias Utilizadas:
Oracle Database SQL (PL/SQL) Tasy EMR Oracle SQL Developer

🚀 Customização de Regras de Negócio no Tasy EMR: Controle de Requisicão de Materiais

1. O que é o projeto

Implementação de uma trava lógica (regra de atributo) no módulo de Requisição de Materiais do ERP Tasy. O objetivo foi condicionar a obrigatoriedade de preenchimento de campos específicos com base no tipo de operação logística selecionada.

2. O Problema Técnico

No fluxo padrão, o campo Atendimento era opcional. No entanto, quando a operação era "Consumo Paciente", o sistema permitia salvar sem vincular um atendimento. Isso gerava quebra na rastreabilidade: o item saía do estoque, mas não era faturado, causando inconsistências financeiras e de inventário.

3. A Solução que aplicamos

Utilizei o dicionário de dados do Tasy para criar uma Regra de Atributo no campo NR_ATENDIMENTO (Tabela REQUISICAO_MATERIAL). Implementei um script SQL de Controle de Obrigatoriedade:

/* Script aplicado no atributo NR_ATENDIMENTO */
AND :CD_OPERACAO_ESTOQUE = [ID_CONSUMO_PACIENTE]

O ícone de obrigatoriedade aparece dinamicamente apenas quando a condição é atendida.

4. O Impacto na Ponta

Garantia de Receita: Eliminou a perda financeira de materiais consumidos e não cobrados. Integridade de Dados: Assegurou 100% de rastreabilidade nos itens de "Consumo Paciente".

🛠️ Tecnologias Utilizadas:
Oracle SQL Dicionário de Dados Tasy Gestão de Estoque

📊 Relatório de Eficiência e SLA Transfusional (Hemoterapia)

1. O que é o projeto

Desenvolvimento de um relatório analítico de SLA (Service Level Agreement) para o setor de Hemoterapia para monitorar todo o ciclo de vida de uma transfusão: desde a solicitação médica até a finalização à beira leito (módulo ADEP).

2. O Problema Técnico

Fragmentação dos dados entre módulos Médico, Laboratório e Enfermagem. Havia dificuldade de rastreabilidade por unidade de hemocomponente, dados de coleta nulos e IDs numéricos que dificultavam a interpretação imediata.

3. A Solução Aplicada

  • Joins Complexos: LEFT JOIN entre SAN_RESERVA, SAN_RESERVA_PROD e SAN_TRANSFUSAO.
  • Tratamento de Nulos (NVL): Subqueries para buscar data de coleta na tabela secundária SAN_AMOSTRA_RECEPTOR.
  • Tradução de Domínios: CASE WHEN para converter códigos (Ex: Código 3 = 'Emergência').
  • Cálculos de Tempo: Colunas calculadas para SLA Agência e SLA Enfermagem.

4. O Impacto na Ponta

Identificação de Gargalos: Visão clara de onde ocorrem atrasos. Segurança do Paciente: Monitoramento de tempos críticos em emergências. Auditoria: Rastreabilidade total dos profissionais para acreditações (ONA/JCI).

🛠️ Tecnologias Utilizadas:
Oracle Database SQL (DML/DQL) Tasy EMR Indicadores de SLA
Zebra EPL • Automação Farmacêutica

🚀 Otimização de Etiquetas de Dispensação por Lote (Sistema Tasy)

1. O que é o projeto?

Desenvolvimento e melhoria de etiquetas térmicas de identificação de medicamentos. Transformou etiquetas simples em ferramentas de segurança clínica para rápida identificação de horários e rastreabilidade de lote.

2. O Problema Técnico

A etiqueta anterior não exibia horários específicos da prescrição, não tinha registro visual do horário de atendimento do lote e apresentava conflitos de layout que impediam a leitura correta pelos scanners.

3. A Solução Aplicada

  • SQL Dinâmico: Uso de LISTAGG e ROW_NUMBER para agrupar múltiplos horários em uma única linha.
  • Programação EPL (Zebra): Redimensionamento do código de barras e reorganização de coordenadas X/Y para evitar sobreposição de textos.

4. O Impacto na Ponta

Segurança: Enfermagem visualiza o horário exato vinculado ao lote na embalagem. Eficiência: Aumento de 100% na taxa de leitura dos códigos pelos coletores de dados.

🛠️ Tecnologias Utilizadas:
Oracle SQL Linguagem EPL (Zebra) Analytical Functions Tasy EMR
Relatórios Delphi-based • SQL Avançado

🚀 Otimização de Relatório de Dispensação Farmacêutica (Tasy EMR)

1. O que é o projeto

Otimização do relatório CATE1011 utilizado para conferência de medicamentos à beira do leito no sistema Tasy EMR.

2. O Problema Técnico

Desperdício: Gerava uma folha para cada lote (se um paciente tivesse 5 lotes, eram 5 folhas). Rastreabilidade: Falta de código de barras individual por linha de medicamento. Layout: Desalinhado e ineficiente.

3. A Solução Aplicada

  • Reestruturação Query: Agrupamento por Atendimento (nr_atendimento) em vez de Lote.
  • Barcodes: Inserção de CODE128_C diretamente na banda de detalhes.
  • Otimização Layout: Redução de espaços vazios e criação de campo concatenado Lote / Prescrição.

4. Impacto na Ponta

Economia: Redução de 60% no consumo de papel e toner. Segurança: Bipagem individual de itens na beira-leito, eliminando digitação manual.

🛠️ Tecnologias Utilizadas:
Gerador de Relatórios Tasy Oracle SQL / PL/SQL Code128
